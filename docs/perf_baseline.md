## 二、模型转换链路（自转换模型验证）

**日期**：2026-07-11

**链路**：

```
YOLOv8n.pt (PyTorch) → yolov8n.onnx (FP32) → yolov8n.rknn (FP16)
```

| 阶段 | 格式 | 大小 |
|------|------|------|
| 原始权重 | PyTorch `.pt` | ~6 MB |
| 导出 | ONNX (FP32) | 12.26 MB |
| 转换 | RKNN (FP16) | 7.9 MB |

**压缩比**：ONNX → RKNN 约 **35% 缩小**（FP32→FP16 权重减半 + RKNN 图元数据）。

**模型结构**：

| 项目 | 值 |
|------|------|
| 架构 | YOLOv8n |
| 输入 | `1 × 3 × 640 × 640`（NCHW） |
| 输出 | `1 × 84 × 8400`（单张量，已 DFL 解码） |
| 84 通道 | 0-3: cx, cy, w, h（像素坐标） + 4-83: 80 类置信度（sigmoid） |
| 8400 | 80×80 + 40×40 + 20×20 三尺度网格拼接 |

**自转换模型推理验证**（bus.jpg）：

| 指标 | 基线 (demo model.rknn) | 自转换 yolo.rknn |
|------|------|------|
| 推理耗时 | ~30ms | 81.6ms |
| 检出数 | — | 5（4 person + 1 bus） |

**自转换 vs Demo 模型对比**（同图 bus.jpg）：

| | 检出数 | box[0] | box[1] | box[2] | box[3] | class | score |
|------|------|------|------|------|------|------|------|
| Python | 5 | 210.9 | 241.5 | 283.6 | 507.5 | 0 (person) | 0.8838 |
| C++ | 5 | 210.9 | 241.5 | 283.6 | 507.5 | 0 (person) | 0.8838 |

**C++ 与 Python 后处理交叉验证：坐标、类别、置信度逐位一致。**

## 三、LLM 性能基线 + 采样参数实验

**日期**：2026-07-16（Week 2 Day 3）

**模型**：Qwen2.5-0.5B-Instruct，W8A8 量化，764MB

### 性能测量方法

在 `llm_demo.cpp` 的 callback 和 main 中插入三个计时点：

| 时间戳 | 位置 | 含义 |
|------|------|------|
| `t0` | `rkllm_run()` 之前 | 推理开始 |
| `t1` | callback 首次 `RKLLM_RUN_NORMAL` | 首 token 到达 |
| `t2` | callback `RKLLM_RUN_FINISH` | 生成结束 |

指标计算：
- **TTFT**（首 token 延迟）= `t1 - t0`
- **Token/s**（生成速度）= `token_count / ((t2 - t1) / 1000.0)`

### 性能数据

| 指标 | 数值 |
|------|------|
| TTFT（首次） | ~114-162ms（含模型预热） |
| TTFT（稳定） | ~80-85ms |
| Token/s | ~34（稳定） |

首次推理 TTFT 明显偏高（~2 倍），原因是首次推理时有额外的初始化开销。

### Temperature 采样实验

| 参数 | temp=0.1, top_k=1 | temp=0.8, top_k=1 | temp=1.5, top_k=50 |
|------|:---:|:---:|:---:|
| 输出多样性 | 完全相同 | 几乎相同 | 四种不同风格 |
| Token/s | 35-37 | 32-33 | 13-37（波动大） |
| 特征 | 安全但无聊 | 稳定 | 新鲜但不可控 |

**关键发现**：改 temperature 但 top_k=1 会压死多样性。top_k=1 只看概率最高的 token，temperature 怎么调都没用。

### 双模型内存预算

```
板子总内存: 7.7 GiB
系统占用:   0.6 GiB
可用:       7.0 GiB
YOLOv8n:    ~0.05 GiB
Qwen 0.5B:  ~1.0-1.5 GiB（含权重 + KV Cache）
加载后剩余:  ~5.5 GiB
```

## 四、双模型管线首次跑通：性能快照

**日期**：2026-07-17（Week 2 Day 4，里程碑）

**条件**：

- `vision_llm_demo.cpp`：单进程，YOLO（librknnrt）+ Qwen（librkllmrt）
- 调度方案：两模型**同时 init**（内存共存，合计 ~1.5GiB）、**串行 run**（NPU 排队），全程无冲突
- 输入：bus.jpg

**结果**：

| 阶段 | 数据 |
|------|------|
| YOLO 检出 | 5 objects（person×4 + bus×1，score 0.33-0.88） |
| LLM 输出 | "一辆公交车上坐着四个人。" |
| LLM TTFT | 96ms |
| LLM Token/s | 41.5 |

**幻觉实验：有无 prompt 约束对比**：

| | 无约束 | 加约束 |
|------|------|------|
| 输出 | 编造"玩手机""准备下车" | "一辆公交车上坐着四个人。" |
| 幻觉 | 有 | 无 |
| TTFT | 252ms | 96ms |
| Token/s | 17.5 | 41.5 |

**验证结论**："同 init、串行 run"的 NPU 分时方案可行。加约束 prompt 同时抑制幻觉和提升性能。

## 五、分段计时（管线拆解）

**日期**：2026-07-19（Week 2 Day 5）

**条件**：`vision_llm_demo.cpp` 中在 `rknn_run` 和 `rkllm_run` 前后加 `steady_clock` 计时点。

**结果**：

| 阶段 | 耗时 | 占比 |
|------|------|------|
| YOLO 推理 | 33 ms | 10.5% |
| LLM 推理 | 281 ms | 89.5% |
| **端到端合计** | **~314 ms** | 100% |

**结论**：LLM 占端到端延迟近 90%，YOLO 几乎不拖后腿。后续优化重点在 LLM 侧。

## 六、换图泛化测试

**日期**：2026-07-19（Week 3 Day 1）

**目的**：用多场景图片验证 YOLOv8n + Qwen 管线的泛化能力和边界。

### 测试结果

| 图片 | 场景 | 真实答案 | YOLO 检出 | 漏检 | 误检 |
|---|---|---|---|---|---|
| bus.jpg | 公交街景 | 4人1车 | 5 (4人1车) | — | — |
| people.jpg | 七人合影 | 7人 | 8 (7人1领带) | — | 领带 (0.27) |
| indoor.jpg | 客厅沙发 | 5人 | 6 (4人2沙发) | 小孩×1 | — |
| cat_dog.jpg | 猫狗玩耍 | 1狗2猫 | 3狗 | 猫×2 | 2狗 |
| cars_people.jpg | 拥堵街头 | 27人25车 | 38 (19人17车1火车1公交) | 8人8车 | 火车、公交 |

### YOLO 推理耗时

| 图片 | 检出数 | YOLO 耗时 | LLM 耗时 | 总耗时 |
|---|---|---|---|---|
| bus.jpg | 5 | 33 ms | 281 ms | 314 ms |
| people.jpg | 8 | 33 ms | 598 ms | 631 ms |
| indoor.jpg | 6 | 33 ms | 359 ms | 392 ms |
| cat_dog.jpg | 3 | 33 ms | 440 ms | 473 ms |
| cars_people.jpg | 38 | 59 ms | 673 ms | 732 ms |

### LLM 场景描述

| 图片 | Qwen2.5-0.5B 输出 |
|---|---|
| bus.jpg | "一辆公交车上坐着四个人。" ✅ |
| people.jpg | "有7个人和一个.tie（可能是一个标记或符号）。" ⚠️ 领带误检 |
| indoor.jpg | "四个人坐在沙发上。" ⚠️ 漏小孩 |
| cat_dog.jpg | "一组被智能摄像头捕捉到的狗图像。" ❌ 全判为狗 |
| cars_people.jpg | "有19个人、17辆汽车和一辆火车。" ❌ 计数严重偏低 |

### 失败分析

YOLOv8**n** 是 nano 版本，COCO mAP 仅 37.3%，以下问题属于**速度-精度 design tradeoff**：

| 问题 | 原因 | 解法 |
|---|---|---|
| 小孩漏检 | 小目标特征弱，n 版本感受野不足以捕获 | 换 YOLOv8s（延迟 ~60ms） |
| 猫→狗混淆 | 同科动物特征相似，n 版本分类头容量小 | 同上，或加针对性 fine-tune |
| 密集场景漏检 | NMS 阈值 + 小目标被大目标抑制 | 调低 conf_threshold，或用 Soft-NMS |
| 领带幻觉 (score 0.27) | 低置信度误检 | 设 score 阈值 ≥ 0.35 即可过滤 |

**结论**：管线在简单到中等场景表现良好（bus/people/indoor），密集遮挡场景到达模型能力边界（cars_people）。这是 nano 版本在 6 TOPS NPU 上的合理表现，并非代码或 pipeline 问题。
