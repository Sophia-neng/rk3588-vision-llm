## 一、YOLO 检测管线

### Q1：YOLOv8n 的输入输出是什么？

**输入**：`1 × 3 × 640 × 640`（NCHW，RGB，归一化到 0-1）

**输出**：`1 × 84 × 8400`（单张量，已 DFL 解码）
- 84 通道：前 4 通道 = cx, cy, w, h（**像素坐标**，不是网格偏移），后 80 通道 = COCO 80 类置信度（已过 sigmoid）
- 8400 = 80×80 + 40×40 + 20×20（三尺度检测头网格拼接）

**关键差异**：Ultralytics 导出的 ONNX 已在内部完成 DFL 解码 + stride 倍乘，输出直接是像素坐标。旧版 demo 模型输出 9 个张量（未解码），这是 RKNN Toolkit 版本差异导致的。

### Q2：预处理做了什么？

1. `imread` 读图 → 2. 计算缩放比（保持宽高比，长边 640）→ 3. `resize` → 4. 灰色填充到 640×640（letterbox）→ 5. BGR→RGB → 6. HWC→CHW → 7. 除以 255 归一化 → 8. `malloc` 分配连续 float buffer → 9. 传入 NPU

### Q3：后处理五步结构是什么？

**cxcywh → xyxy**（中心坐标转角点）→ **找最高分类别**（argmax 80 通道）→ **过滤低分框**（score > threshold）→ **坐标裁剪到图像边界** → **NMS 去重**（IoU 阈值筛选）

### Q4：NMS（非极大值抑制）的原理？

按置信度降序排列所有候选框 → 取最高分的框加入保留集 → 计算它和剩下所有框的 IoU → 删掉 IoU 超过阈值的重叠框 → 重复直到候选集清空。**一句话：同一物体只保留置信度最高的那个框。**

### Q5：为什么 8400 个候选框是三个尺度？

YOLOv8 有三个检测头：80×80（浅层，检测小物体，网格最密）、40×40（中层，检测中等物体）、20×20（深层，检测大物体，感受野最大）。三尺度输出拼成一个张量，不需要手动拆开——后处理时 8400 个候选框统一处理。

## 二、LLM 部署概念

### Q6：什么是 Token？

LLM 不认字，只认数字。**Tokenizer 把文本切成一个个最小单元（token），每个 token 映射为一个整数 ID。** 中文一个字可能是一个 token，英文一个单词可能拆成多个 token。Qwen2.5 的 tokenizer 有约 15 万词表。

### Q7：什么是 Embedding？

把 token ID 映射成一个**高维稠密向量**（比如 896 维）。语义相近的词向量相近。这是 LLM 理解的"词义"。

### Q8：什么是自回归（Autoregressive）生成？

LLM 一次只生成一个 token。流程：输入 "今天的天气" → 输出 "真" → 把 "真" 拼回去变成 "今天的天气真" → 输出 "好" → 循环，直到输出结束符或达到 max_tokens。**每一步的输出变成下一步的输入。**

### Q9：什么是 KV Cache？为什么重要？

自回归每步都要对**全部历史 token** 做 Attention。如果没有 KV Cache，第 100 步要重算前 99 步的 Key/Value，计算量 O(n²)。**KV Cache 把每步算好的 K/V 存起来，新步只算新 token 的 Attention**，降到 O(n)。

**对端侧部署的意义**：KV Cache 是 LLM 推理的主要内存占用之一。2048 context × 0.5B 参数 ≈ 几百 MB。

### Q10：W8A8 量化是什么意思？

**权重（Weight）8-bit 整数 + 激活值（Activation）8-bit 整数。** 原始 FP16 模型 → RKLLM Toolkit 转换 → 权重和中间结果都用 INT8 存和算。好处：内存减半、NPU 友好（RK3588 NPU 对 INT8 有专用加速单元）。代价：微小精度损失。Qwen2.5-0.5B：FP16 ~1.5GB → W8A8 ~764MB。

### Q11：0.5B 参数是什么意思？为什么选这个规模？

B = Billion = 10 亿。0.5B = 5 亿参数。Qwen2.5-0.5B 是端侧能跑的最小可用模型之一。**选这个规模的原因**：8GB 板端内存能同时装下 YOLO + LLM 两个模型（合计 ~1.5GiB），参数量再大内存不够、速度太慢。0.5B 的 TTFT ~96ms、生成速度 ~41 tok/s，够做场景描述。

### Q12：什么是 LLM 幻觉（Hallucination）？怎么抑制？

LLM 的本质是**自回归续写最像人话的下一个词**，不是在报告事实。输入 "4 person, 1 bus"，训练数据里 "公交车" 后面常跟 "乘客玩手机"——它就顺着续写。temperature=0.1 也压不住。

**解决方案**：Prompt 设计 = **指令 + 数据 + 约束**。"只根据给出的物体列表描述，不要添加列表之外的细节。" 效果：无约束 TTFT 252ms/17.5 tok/s → 加约束 96ms/41.5 tok/s（输出更短更快）。

## 三、RKNN 工具链

### Q13：模型转换链路是什么？

```
PyTorch (.pt) → ONNX (.onnx, FP32) → RKNN (.rknn, FP16)
```

工具：`torch.onnx.export` → `RKNN Toolkit2`（`rknn.build` 或 `rknn.export_rknn`）

### Q14：RKNN C API 的 6 个核心函数？

| 函数 | 作用 |
|------|------|
| `rknn_init(&ctx, path, ...)` | 加载模型到 NPU |
| `rknn_query(ctx, IN_OUT_NUM, ...)` | 查询输入/输出数量 |
| `rknn_inputs_set(ctx, 1, inputs)` | 喂输入数据 |
| `rknn_run(ctx, NULL)` | 触发 NPU 推理 |
| `rknn_outputs_get(ctx, 1, outputs)` | 取输出数据 |
| `rknn_destroy(ctx)` | 释放 NPU 资源 |

Python 的 `model.run()` 内部封装了 set → run → get 三步。

### Q15：FP16 vs INT8 量化的区别？

| | FP16 | INT8 |
|------|------|------|
| 精度 | 几乎无损 | 微小损失 |
| 文件大小 | ~ONNX 的 50% | ~ONNX 的 25% |
| 速度 | 快 | 更快（NPU 有 INT8 专用加速） |
| 额外工作 | 零（RKNN Toolkit2 默认） | 需要校准数据集 ~100 张 |

Week 1 做 FP16 基线，INT8 是后续对比实验。

### Q16：LLM 推理为什么是回调驱动的？

`rkllm_run()` 不是同步返回结果，而是每生成一个 token 调用一次 callback。原因是 LLM 生成是流式的——你不知道要生成多少个 token。**回调模式让上层可以边生成边处理（打印、streaming、计时）。** callback 收到 `RKLLM_RUN_FINISH` 标志时表示生成结束。

### Q17：RKLLM 的三个关键生成参数？

| 参数 | 含义 | 低值效果 | 高值效果 |
|------|------|------|------|
| temperature | 缩放概率分布 | 选安全词，保守 | 给冷门词机会，多样 |
| top_k | 候选集大小 | 1=贪心，只选最高概率 | 50=从 top-50 里采样 |
| repeat_penalty | 重复惩罚 | 允许重复 | 抑制已出现过的词 |

**踩坑**：top_k=1 会压死 temperature 效果——不管 temperature 设多少都只选最高概率那个词。先放开 top_k 再调 temperature。

## 四、C++ 关键概念

### Q18：`c_str()` 的悬空指针问题？

`c_str()` 返回 `std::string` **内部内存**的指针，不是拷贝。之后对 string 的任何修改（`+=`、`append`、`resize`）都可能触发扩容，旧指针指向已释放的内存。**规矩：字符串定稿之后才取 `c_str()`。**

### Q19：`-L` 和 `-l` 的区别？

`-L` 指定**库搜索目录**（链接器去哪找 .so 文件），`-l` 指定**库名**（链接器自动补 `lib` 前缀和 `.so` 后缀）。`-L/home/lineng -lrknnrt` → 链接器在 `/home/lineng` 找 `librknnrt.so`。**-L 给目录，-l 给库名，各管各的。**

### Q20：NCHW 内存布局和 anchor-major 的区别？

NPU 输出的 `(1,84,8400)` 是 NCHW（channel-major）布局：`[ch0 的 8400 个值] [ch1 的 8400 个值] ... [ch83 的 8400 个值]`。但后处理需要 anchor-major：`[anchor0 的 84 个值] [anchor1 的 84 个值] ...`。**必须做一次转置**，否则读到的数据是乱的（同一个 bug 踩了两次）。

### Q21：C++ 后处理为什么容易出错？

Python 的 `boxes[mask]` 布尔索引一行搞定，C++ 要 `for + if + push_back` 手动筛。Python 的 `np.maximum` 对应 C++ 的 `std::max`，但语义不是一一对应的。**Python 和 C++ 之间不是语法翻译，是思路翻译。** 编译器救不了逻辑错误——编译通过 ≠ 结果正确。

### Q22：vector 出参为什么要加 `&`？

C++ 默认传值——函数里改的是副本，调用者拿到的还是空的。加 `&` 是传引用，改的是原对象。**这是 C++ 初学者最容易无声翻车的地方：编译不报错，程序不崩溃，只是结果为空。**

## 五、系统设计

### Q23：为什么两个模型可以放在同一个进程里跑？

RK3588 NPU 是**分时复用**的，不是分区独占。两个模型同时初始化（各自加载权重到内存），推理时排队串行跑。内存不是瓶颈——YOLO ~50MB + Qwen ~1GB，合计 ~1.5GiB，8GB 总内存剩余 5.5GiB。

### Q24：NPU 分时方案的设计思路？

当前方案（已验证）：**同时 init，串行 run。** 两个模型内存共存、推理排队，无冲突。

下一步（Week 3）：**时间分片 + 流水线并行**。摄像头帧到达 → YOLO 推理 → 结果立刻送给 LLM → 同时 YOLO 开始处理下一帧。目标是 YOLO 和 LLM 的推理时间重叠，提高吞吐。

### Q25：双模型管线的完整数据流？

```
摄像头/图片 → 预处理(resize+letterbox+归一化) → NPU(YOLO推理) → 后处理(解码+NMS)
→ 物体列表("4 person, 1 bus") → 构建prompt(指令+数据+约束) → NPU(LLM推理)
→ 逐token回调 → 场景描述("一辆公交车上坐着四个人。")
```

### Q26：为什么选择 YOLOv8n + Qwen2.5-0.5B 这个组合？

**YOLOv8n**：最小的 YOLOv8 变体，检测精度够用，推理 ~30ms。**Qwen2.5-0.5B**：端侧能跑的最小可用 LLM，W8A8 压缩后 764MB。两个模型加起来能在 8GB 板端内存共存。这个组合证明了"端侧视觉理解"的可行性下限——两三年后模型更小更快，方案会更成熟。

## 七、踩坑故事（展示真实经验）

### 故事 1：输出格式差点让我重写整个后处理

demo 模型输出 9 个张量，我自己转换的只输出 1 个。一开始以为是转错了，反复查才发现是 RKNN Toolkit 版本差异——新版默认合并输出。更坑的是坐标格式：我按 demo 代码的 ltrb 网格偏移来写后处理，dump 数据发现值不对，才发现 Ultralytics 导出已经做了 DFL 解码 + stride 倍乘，输出直接是 cxcywh 像素坐标。

**面试官想听**：你遇到矛盾时不是猜，是 dump 实际数据验证。

### 故事 2：c_str() 悬空指针，编译 0 error 照样错

代码编译通过、不崩溃、输出也正常——但 LLM 有时候输出乱码。排查发现 `c_str()` 写在了 prompt 拼接循环之前，之后每次 `+=` 都可能让 string 扩容搬家。改完只挪了两行代码，但定位花了不少时间。

**面试官想听**：你知道 C++ 的内存模型，不是只会写能编译的代码。

### 故事 3：幻觉实验 — prompt 一句话让速度翻倍

无约束时 LLM 输出 "有四个人在一辆公共汽车上，其中一人正在使用手机进行社交互动"——全是编的，TTFT 252ms。加一句 "只根据物体列表描述" 后输出变成 "一辆公交车上坐着四个人。" 零幻觉，TTFT 96ms（快了 2.6 倍）。**小模型的 prompt engineering 不是锦上添花，是刚需。**

### 故事 4：g++ 命令断行 — 没报错 ≠ 编译成功

终端只回了一句 `-O2: command not found`，扫一眼没有 "error" 字样就以为过了。结果 `ls` 发现根本没产物——开头 `g++ -` 被截掉了，shell 把 `-O2` 当成命令名执行。**教训：验收标准是产物存在，不是"没看到 error"。**

## 八、高频追问

### Q：为什么不直接用 ChatGPT API，要在板端跑？

端侧推理三大优势：**低延迟**（不经过网络）、**隐私**（数据不出设备）、**离线可用**（不依赖服务器）。安防摄像头、车载、机器人这些场景网络不可靠，端侧是刚需。

### Q：如果给你更多时间，下一步优化什么？

**流水线并行**：YOLO 推理和 LLM 推理时间重叠，提高吞吐。**INT8 量化**：进一步压缩模型、加速推理。**零拷贝推理**：用 `rknn_create_mem` 直接分配 NPU 物理内存，省掉 CPU↔NPU 之间的 memcpy。

### Q：你为什么做这个项目？

边缘 AI 部署是我想深耕的方向——把模型从服务器搬到嵌入式设备上，需要同时懂模型（精度）、懂硬件（NPU 特性）、懂系统（内存/调度）。这个项目覆盖了这三个层面，从 ONNX 转换到 NPU 推理到双模型协同，是我对端侧部署理解的集中体现。

## 九、水下图像增强（Retinex + CLAHE 多路径增强系统）

> 时间：2025.5 – 2025.7　　IEEE ICIPCA 2025 录用

### Q1：这个项目解决了什么问题？

水下图像退化严重——水对光的吸收和散射导致**颜色失真、对比度低、细节模糊**。单一增强算法（比如只用 CLAHE 拉对比度）对不同退化类型的鲁棒性差。这个项目设计了一套**多路径增强系统**，根据退化类型自动选择合适的增强策略。

### Q2：Retinex 理论是什么？

人眼/相机看到的图像 = **反射分量（reflectance） × 光照分量（illumination）**。Retinex 的核心思想：**从原图中估计并去除光照分量，保留物体本身的反射分量**，从而消除光照不均匀的影响。

水下场景中，光照分量受水深、水质、光源角度影响剧烈。Retinex 把光照和物体颜色解耦，单独矫正光照，这比直接在 RGB 上调对比度更符合物理规律。

### Q3：CLAHE 是什么？和普通直方图均衡化有什么区别？

**CLAHE = Contrast Limited Adaptive Histogram Equalization**（对比度受限的自适应直方图均衡化）。

- 普通直方图均衡化（HE）：全局拉伸灰度分布 → 噪声也被放大
- 自适应（AHE）：分小块各自均衡 → 块边缘有伪影
- **CLAHE**：AHE + **对比度裁剪**（clip limit，把直方图超出阈值的部分均匀分回去）→ 既增强局部对比度又不放大噪声

水下图像适合 CLAHE 因为退化是空间不均匀的——近处清楚远处模糊，全局均衡化无效。

### Q4：Retinex 和 CLAHE 在同一个系统里怎么分工？

两者互补：
- **Retinex**：解决**光照不均匀**（去光照分量），恢复颜色真实性
- **CLAHE**：解决**局部对比度低**（分块均衡），恢复纹理细节

多路径设计：根据物理退化模型检测退化类型（是散射主导还是吸收主导），不同退化走不同的增强路径——Retinex 为主 + CLAHE 辅助，或 CLAHE 为主 + Retinex 后处理。

### Q5：物理退化模型里的三种散射分量是什么？

水下图像退化不只是"变模糊"，光源→物体→相机之间有三段光路：

| 分量 | 光路 | 影响 |
|------|------|------|
| **直接分量** | 物体反射光直接到达相机 | 有用信号 |
| **后向散射** | 光源打到水中悬浮粒子，散射进相机 | 雾状噪声，降低对比度 |
| **前向散射** | 物体反射光被粒子散射偏离、再散射进入 | 模糊，边缘退化 |

三种分量合在一起构建的退化模型比简单的"加雾"更接近真实水下光学。这也是为什么论文能被录——不是纯调参，有物理建模。

### Q6：多任务 CNN 做什么？

输入一张水下图像，CNN 同时做两件事：**判断退化类型**（散射型/吸收型/混合型）+ **估计退化程度**。输出控制后面的增强路径选择和参数设置。多任务设计的好处：两个任务共享浅层特征（纹理、颜色分布），比两个独立模型更高效。

### Q7：面试官追问"你的贡献是什么？"

三个层次回答：
1. **物理建模**：不是复现别人的增强算法，而是从散射物理模型出发设计了退化框架
2. **多任务 CNN**：自动检测退化类型，让增强策略自适应（而非手动调参）
3. **系统验证**：在真实水下图像数据集上验证，有论文发表（IEEE ICIPCA 2025，EI 收录）

### Q8：为什么水下图像增强不能用常规图像增强方法？

三个原因：
1. **退化非均匀**：不同水深、水质、光照条件下退化机制完全不同。Instagram 滤镜式的全局增强不管用
2. **颜色通道退化不对称**：红光在水下衰减最快（水吸收红光），蓝绿光穿透更深 → 水下照片偏蓝绿。常规方法不建模通道间的差异
3. **物理约束**：增强是为了后续任务（目标检测、分类），恢复必须忠实于物体真实颜色，不能"好看但不对"

### 以下为深入版（不需要背，面试前扫一眼即可）

### Q1：这个项目测量什么？为什么难？

测量直径 **< 0.3mm**、深径比 **> 3** 的微孔三维形貌。难点：孔太小（0.3mm ≈ 3 根头发丝并排）、太深（深径比 >3 → 光进不去出不来）、**常规显微镜只能看到表面，看不到孔内壁**。

### Q2：条纹投影（Fringe Projection）的原理是什么？

投影仪把一组**正弦条纹图案**投到被测物体表面 → 物体表面的三维形状会扭曲条纹（高度不同 → 相位偏移不同）→ 相机拍下变形条纹 → 从相位偏移反推三维高度。

一句话：**用条纹的变形量编码表面高度**，类似"用影子测凹凸"但精度高得多。

### Q3：为什么用 Gray 码？什么是相位跳变？

条纹投影的问题：相位是周期性的（0→2π→0→2π...），相机记录的相位在 0 到 2π 之间。工件表面高度超过一个周期时，相位直接从 2π 跳回 0——这叫**相位跳变**，导致高度计算错误。

**Gray 码**是一种二进制编码图案，相邻数只变一位。投影多幅 Gray 码图案 → 每个像素得到一个唯一的二进制码 → 告诉相位仪"你现在在第几个周期" → 展开成连续相位（phase unwrapping）。

### Q4：为什么还要加"互补码"双验证？

Gray 码展开有一个经典 bug：**条纹边缘处**（黑白交界），正弦条纹的相位和 Gray 码的二进制边界可能**错开一个像素**，导致展开后的相位跳错一个周期。

互补码就是 Gray 码整体平移半个周期再投一次，两次结果互相校验——不一致的地方就是边界错误。**Gray 码告诉你第几个周期，互补码确认周期编号没错。**

### Q5：整个测量流程是怎么串起来的？

```
虚拟微孔建模(CAD) → 生成条纹图案 → 投影仪投射到真实微孔
→ 双目相机采集变形条纹 → Gray码+互补码相位展开
→ 双目立体匹配(左右相机同名点) → 三角测量求深度 → 三维点云重建
```

先仿真验证算法正确性，再上硬件跑真实样件。

### Q6：为什么用双目视觉而不是单目？

单目只能从一张图推断深度（靠阴影、纹理等线索），不精确。**双目视觉靠三角测量**：同一个物理点在左右相机成像位置不同（视差），视差 × 基线距离 → 精确深度。对于微孔这种要求微米级精度的场景，双目是必需的。

### Q7：硬件加速做了什么？

Gray 码投影需要多幅图案（一般 7-8 幅），加上互补码就是 14-16 幅。每幅曝光 + 传输 + 处理需要时间。硬件加速：投影仪和相机用硬件触发同步 → 缩短帧间等待 → 测量时间从分钟级降到秒级。

### Q8：面试官追问"你的贡献是什么？"

1. **微孔专用装置设计**：常规条纹投影测的是宏观物体（人脸、零件），你针对 <0.3mm 微孔做了光学系统适配（镜头倍率、投射角度、光照控制）
2. **Gray 码 + 互补码双验证**：不是照搬书上相位展开方法，加入了互补码作为错误检测机制，解决微孔深径比大导致的投影阴影和边界错误
3. **全流程覆盖**：从仿真到硬件再到算法，不是"帮师兄跑了一部分"，而是独立完成完整链路
4. **竞赛验证**：华北赛区二等奖，说明方案经受了同行评审

## 十一、C++ 高频基础

> 这些是 C++ 面试最高频的问题，和项目关系不大但必问。每个答 1-2 分钟即可。

### Q1：unique_ptr 和 shared_ptr 的区别？什么时候用哪个？

**一句话**：`unique_ptr` 独占所有权，`shared_ptr` 共享所有权。

`unique_ptr`：同一时刻只有一个指针拥有对象。不能拷贝，只能移动（`std::move`）。离开作用域自动释放。开销几乎为零。适用于明确只有一个所有者的场景——比如工厂函数返回值、容器里的对象。

`shared_ptr`：多个指针共享同一个对象，内部有**引用计数**。每复制一个 +1，每析构一个 -1，计数归零时释放对象。有额外内存开销（控制块存引用计数）。适用于多个地方需要访问同一对象的场景——但能用 `unique_ptr` 就不要用 `shared_ptr`。

**选型口诀**：优先 `unique_ptr` → 确实需要共享才用 `shared_ptr` → 配合 `weak_ptr` 打破循环引用。

### Q2：虚函数是什么？虚函数表怎么工作的？

**一句话**：虚函数让父类指针/引用在运行时调用子类的重写版本，而不是编译时绑定父类版本。

**虚函数表（vtable）**：每个有虚函数的类有一张函数指针表，存在只读数据段。每个该类的对象头部有一个隐藏指针（vptr）指向这张表。运行时调用虚函数 → 通过 vptr 找到 vtable → 查表跳到真正的函数地址。

**关键**：
- 构造函数不能是虚函数（对象还没建完，vptr 没初始化）
- 析构函数应该是虚函数（不然父类指针 delete 子类对象时只调父类析构，泄漏子类资源）
- 虚函数有额外开销：一次指针跳转 + 每个对象多一个 vptr（8 字节）

### Q3：RAII 是什么？为什么 C++ 这么强调它？

**一句话**：资源获取即初始化——把资源的生命周期绑到对象的生命周期上。

构造函数里拿资源（`new`、`fopen`、`lock`），析构函数里还资源（`delete`、`fclose`、`unlock`）。对象离开作用域时析构函数**自动**被调用，不会忘记释放。即使 `return`、抛异常、`break` 都不会漏。

**反面教材**：C 风格的 `malloc/free`——你必须在每个出口手动 `free`，漏一个就是内存泄漏。RAII 把这个负担从"人记住"变成了"编译器记住"。

**C++ 里最常见的 RAII**：`std::vector`（析构释放内存）、`std::lock_guard`（析构解锁）、`std::ifstream`（析构关闭文件）。

### Q4：移动语义（move semantics）解决了什么问题？

**一句话**：把"深拷贝"变成"转移所有权"，避免不必要的复制。

传统拷贝是一个字节一个字节复制，一个大 vector 拷贝很贵。但很多场景里源对象马上就不要了——比如函数返回局部变量、`push_back` 临时对象。这时直接把源对象的内部指针"偷"过来，省掉内存分配和数据复制。

**右值引用 `&&`**：标记"这个对象可以被偷"。`std::move` 把左值转成右值引用，告诉编译器"这个变量我不用了，你拿走吧"。移动后源对象处于"有效但未指定"状态——可以析构或重新赋值，不要再读它。

**实际收益**：函数返回一个大 vector 时，移动语义把 O(n) 的拷贝变成 O(1) 的指针交换。

### Q5：const 的三种用法？

| 位置 | 含义 |
|------|------|
| `const int* p` | 指向的值不能改（底层 const） |
| `int* const p` | 指针本身不能改（顶层 const） |
| `void foo() const` | 成员函数不修改成员变量 |

最后一个（const 成员函数）口诀：**const 后面表示这个函数对 `this` 的承诺——我不改你。** 面试常考：const 对象只能调 const 成员函数。

### Q6：vector 扩容机制？为什么用 emplace_back 而不是 push_back？

**扩容**：当 `size() == capacity()` 时再插入会触发扩容——申请一块更大的内存（通常是 1.5 倍或 2 倍），把旧元素**移动**过去，释放旧内存。扩容的均摊复杂度是 O(1)，但单次扩容是 O(n)。

**push_back vs emplace_back**：
- `push_back(x)`：先在外面构造对象 x → 再拷贝/移动到 vector 里 → 一次构造 + 一次拷贝/移动
- `emplace_back(args...)`：直接把参数转给 vector，**在 vector 内部原地构造**→ 只有一次构造，零拷贝

**口诀**：往 vector 里放新对象用 `emplace_back`，放已有对象用 `push_back`。

## 十二、深度学习基础

> 边缘 AI 部署岗位，面试官默认你懂这些。不需要推导公式，能讲清楚"是什么、为什么、怎么用"就行。

### Q1：CNN 的几层各干什么？

| 层 | 作用 | 一句话 |
|------|------|------|
| 卷积层 Conv | 提取局部特征（边缘、纹理、形状） | 用一个小滑动窗口扫全图 |
| 池化层 Pooling | 降采样，减少计算量 | 最大池化：每个区域只保留最大值 |
| 激活函数 ReLU | 引入非线性 | max(0, x)，负数变零 |
| 全连接层 FC | 把特征综合成分类结果 | 每个输入连到每个输出 |
| BatchNorm | 稳定训练，加速收敛 | 每批数据归一化到均值为 0 方差为 1 |

### Q2：为什么用 ReLU 而不是 Sigmoid？

**Sigmoid 的问题**：x 很大或很小时梯度趋近于 0（梯度消失），深层网络前面几层几乎不更新。Sigmoid 输出不是零中心的，影响优化效率。

**ReLU 的优势**：x>0 时梯度恒为 1，不衰减，深层网络也能传回梯度。计算量极小（就是一个 `if x>0`）。实践中收敛比 Sigmoid 快得多。

**ReLU 的缺点**：x<0 时梯度为 0，某些神经元可能"死掉"（永远输出 0）。变体 Leaky ReLU 在负数区给一个小斜率解决这个问题。

### Q3：Batch Normalization 做了什么？为什么有效？

**做什么**：每一层的输入在 batch 内归一化到均值 0、方差 1，再乘一个可学习的缩放因子加一个偏移。

**为什么有效**：
- 每层输入分布稳定 → 可以用更大的学习率 → 训练更快
- 轻微的正则化效果（batch 统计量有噪声，相当于加了噪声）
- 减少对初始化权重的敏感度

**推理时**：没有 batch 这个概念，用训练时存下来的全局均值和方差代替。

### Q4：过拟合是什么？怎么解决？

**现象**：训练集上准确率很高，测试集上很差——模型背下了训练数据而非学到规律。

**解决方案**（按优先级）：
1. **更多数据**：数据增强（翻转、裁剪、加噪），最有效
2. **Dropout**：训练时随机关闭一部分神经元（不参与前向和反向），强迫网络不要依赖单个神经元
3. **正则化**：L1（权重绝对值之和）鼓励稀疏，L2（权重平方和）鼓励权重小且均匀
4. **早停 Early Stopping**：验证集 loss 不再下降就停
5. **更简单的模型**：减少层数或神经元数

### Q5：梯度下降的基本原理？

**一句话**：沿着 loss 函数下降最快的方向（负梯度方向）调整参数，一步步走到最低点。

参数 = 参数 - 学习率 × 梯度。学习率控制了每一步迈多大——太大可能跳过头不收敛，太小收敛太慢。

**三种变体**：
| 类型 | 数据量 | 特点 |
|------|------|------|
| Batch GD | 全量数据 | 最准但最慢 |
| SGD | 1 个样本 | 最快但最抖 |
| Mini-batch GD | 一小批（32/64） | 折中，实际最常用 |

### Q6：为什么模型部署要用 ONNX 作为中间格式？

**一句**：框架互操作的通用语言。PyTorch 训模型、TensorFlow 训模型、paddle 训模型 → 各自导出 ONNX → 推理引擎（RKNN/TensorRT/OpenVINO）统一吃 ONNX。

本质是**解耦训练和推理**。训练框架百花齐放，推理引擎各厂商自研，ONNX 充当桥梁——一次导出，到处部署。ONNX 标准化了算子定义和图结构表示，不是代码而是一种模型表达格式（protobuf）。

### Q7：FP16 量化为什么能减小模型体积？精度为什么没丢太多？

**体积减半**：每个权重从 32 位（FP32）变成 16 位（FP16），存储直接减半。运行时内存也减半。

**精度几乎不丢**：神经网络推理是大量矩阵乘加——这些操作对低精度不敏感。权重值的精确到小数点后六位和三位，对最终结果的差异远小于其他噪声（初始化、数据采样）。FP16 的指数范围也够覆盖权重和激活值的典型分布。

**INT8 损失略大**：INT8 只有 256 个离散值，需要校准数据集确定缩放因子。但对大多数 CV 任务，精度损失 <1%，而速度提升和内存节省显著。

### Q6：你还有什么要问我的？（反问面试官）

**一定要问！问 2-3 个。不问 = 不感兴趣。**

推荐问题：
1. **"团队目前用的是什么推理框架/芯片平台？"** — 展示你关心技术栈
2. **"新人入职后前三个月一般做什么？有什么培养机制？"** — 务实不装

## 十四、Linux/嵌入式基础

> 边缘 AI 部署岗大概率会问，但不会很深。记住核心概念即可。

### Q1：进程和线程的区别？

| | 进程 | 线程 |
|------|------|------|
| 资源 | 独立内存空间，不共享 | 共享进程内存 |
| 通信 | 管道、消息队列、共享内存（重） | 直接读写共享变量（轻） |
| 创建开销 | 大（复制页表） | 小 |
| 崩溃影响 | 一个崩不影响其他 | 一个崩可能带崩整个进程 |

**记忆**：进程是资源分配的最小单位，线程是 CPU 调度的最小单位。一个进程可以有很多线程，它们共享同一片内存但各有各的栈。

### Q2：互斥锁（mutex）和条件变量（condition_variable）分别解决什么问题？

**mutex**：保护共享数据，同一时刻只有一个线程能访问——解决**互斥**问题。C++ 用 `std::mutex` + `std::lock_guard`（RAII 自动解锁）。

**condition_variable**：线程间通知——解决**同步**问题。一个线程等条件（`wait`），另一个线程满足条件后通知（`notify_one`/`notify_all`）。生产者-消费者就是经典例子。

**注意**：`wait` 必须配合 mutex 使用，wait 内部会**释放锁然后阻塞**，被唤醒后**重新获取锁**再返回。

### Q3：编译的四个阶段是什么？

```
预处理 → 编译 → 汇编 → 链接
```

| 阶段 | 输入 | 输出 | 做什么 |
|------|------|------|------|
| 预处理 | `.c/.cpp` | `.i`（展开后源码） | 展开 `#include`、替换 `#define`、去掉注释 |
| 编译 | `.i` | `.s`（汇编代码） | 源码 → 汇编，语法检查在这里 |
| 汇编 | `.s` | `.o`（目标文件） | 汇编 → 机器码，符号未解析 |
| 链接 | `.o` + `.a/.so` | 可执行文件 | 解析符号引用，合并段 |

**面试常考**：头文件在预处理阶段展开，`-I` 影响预处理搜索路径；`-L`/`-l` 影响链接阶段。

### Q4：什么是交叉编译？

**一句话**：在一个平台上编译出另一个平台上运行的程序。

比如在 Mac（ARM64）上编译出 Orange Pi（aarch64 Linux）的可执行文件。需要交叉编译工具链（cross-compiler）和目标平台的库。

**为什么需要**：嵌入式设备算力弱、存储小，编译 OOM 或极慢。所以在开发机上交叉编译，产物 scp 到板端。

**你的实践**：`g++ aarch64` 直接在板端编译。如果做交叉编译，可以用 `aarch64-linux-gnu-g++` 在 Mac/Ubuntu 上编译然后传过去。

### Q5：介绍一下你在 Orange Pi 上做过的 Linux 操作

**这不是技术问题，是验证你是否真的用过 Linux。**

> 我日常通过 SSH 连接 Orange Pi 进行开发。系统用的是 Ubuntu 22.04 ARM 版。基本操作包括：`scp` 传输模型和代码、`g++` 编译 C++ 管线、配置 `LD_LIBRARY_PATH` 让程序找到 `.so` 动态库、`htop` 查看 CPU/NPU 占用、`dmesg` 检查 NPU 驱动加载状态。也配置过 `nmcli` 多 Wi-Fi 自动切换和 mDNS 域名解析。模型转换部分我用了 Colima 跑 x86 Docker 容器来执行 RKLLM Toolkit，因为转换工具没有 ARM 版。
